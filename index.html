Frigate NVR
Proxmox Setup Guide

A summary guide on how I set up Frigate NVR on Proxmox with Docker, NVIDIA GPU acceleration, multi-camera support, Zabbix monitoring (optional), and automated backups.

Proxmox VE 8.3.0 Frigate 0.17.1 NVIDIA GTX 1050 Ti Driver 580.142 2× H.265 Cameras 1TB Storage
01

System Overview

Frigate is an open-source Network Video Recorder (NVR) that uses AI to detect people, vehicles, and other objects in your camera feeds. It runs inside a lightweight Linux container (LXC) on Proxmox, managed by Docker. You do not need prior Linux experience — each step is explained in full with exact commands to run.

Directory Structure

All Frigate files live under /root/ inside LXC CT-100. Understanding this structure helps you know where to edit configs, where recordings are stored, and what to back up.

filesystem
/root/
├── docker-compose.yml       # Docker configuration
├── frigate.db               # Persistent database (alerts/detections)
├── config/
│   └── config.yml           # Frigate configuration
└── media/
    └── records/             # Loop1 1TB recording storage
        ├── recordings/      # Continuous recording segments
        ├── clips/           # Event clips and thumbnails
        └── exports/         # Exported footage
⚠️
Important frigate.db must exist as a file before starting the container. Run touch ~/frigate.db to create it. If Docker creates it as a directory first, you'll get a mount error.

Storage Configuration

Frigate needs two types of storage: a small root disk for the OS and configs, and a large disk for recording footage. This setup uses Proxmox loop devices (virtual disks) for both.

Proxmox Mount Points

DeviceSizeMount PointPurpose
loop020GB/Root filesystem
loop11TB/media/recordsRecording storage
proxmox config
cctv-records:100/vm-100-disk-1.raw,mp=/media/records,backup=1,size=1000G

docker/docker-compose.yml

yaml
services:
  frigate:
    container_name: frigate-cctv
    restart: unless-stopped
    stop_grace_period: 30s
    image: ghcr.io/blakeblackshear/frigate:stable-tensorrt
    runtime: nvidia
    shm_size: "1024mb"            # 1GB shared memory for 2+ cameras
    environment:
      - NVIDIA_VISIBLE_DEVICES=all
      - NVIDIA_DRIVER_CAPABILITIES=all
      - TZ=Asia/Manila              # timezone
    devices:
      - /dev/nvidia0:/dev/nvidia0
      - /dev/nvidiactl:/dev/nvidiactl
      - /dev/nvidia-uvm:/dev/nvidia-uvm
    volumes:
      - ./config:/config                    # config persistence
      - /media/records:/media/frigate        # 1TB recording storage
      - ./frigate.db:/config/frigate.db     # database persistence
      - /etc/localtime:/etc/localtime:ro    # sync timezone
      - /etc/timezone:/etc/timezone:ro      # sync timezone name
      - type: tmpfs
        target: /tmp/cache
        tmpfs:
          size: 2000000000               # 2GB in-memory segment cache
    ports:
      - "8971:5000"
      - "8554:8554"

Frigate Configuration

The Frigate configuration file (config/config.yml) controls everything — cameras, recording, detection, and GPU usage. It uses YAML format where indentation matters. Always use spaces, not tabs.

💡
After every edit to config.yml, restart Frigate with: docker compose down && docker compose up -d

05

Data Persistence

By default, Docker containers are stateless — all data is lost when the container stops. To keep your recordings, camera settings, and detection history, critical files must be stored outside the container on the host filesystem via Docker volume mounts.

First-Time Setup

bash
# Create config directory
mkdir -p ~/config
nano ~/config/config.yml

# Create database file (MUST be a file, not a directory)
touch ~/frigate.db

# If container already running, copy existing DB first
docker cp frigate-cctv:/config/frigate.db ~/frigate.db

# Start Frigate
docker compose up -d

# Verify everything loaded
docker logs frigate-cctv 2>&1 | grep -i "config\|camera\|database"

Safe Restart Procedure

bash
docker compose down && docker compose up -d
⚠️
If frigate.db doesn't exist as a file, Docker creates it as a directory causing: Are you trying to mount a directory onto a file?
Fix: rm -rf ~/frigate.db && touch ~/frigate.db
06

NVIDIA GPU Setup

Frigate uses your NVIDIA GPU to decode video streams and run AI object detection. Without GPU acceleration, Frigate uses the CPU which is much slower and cannot handle multiple cameras reliably. Follow these steps in order.

Step 1 — Install Driver on Proxmox Host

bash — proxmox host
# Add NVIDIA CUDA repository
wget https://developer.download.nvidia.com/compute/cuda/repos/debian12/x86_64/cuda-keyring_1.1-1_all.deb
dpkg -i cuda-keyring_1.1-1_all.deb
apt update && apt install -y dkms cuda-drivers pve-headers-$(uname -r)

# Blacklist nouveau (open-source driver that conflicts with NVIDIA)
echo "blacklist nouveau" > /etc/modprobe.d/blacklist-nouveau.conf
echo "options nouveau modeset=0" >> /etc/modprobe.d/blacklist-nouveau.conf
update-initramfs -u -k all && reboot

Step 2 — LXC GPU Passthrough

Edit /etc/pve/lxc/100.conf on the Proxmox host:

/etc/pve/lxc/100.conf
lxc.cgroup2.devices.allow: c 195:* rwm
lxc.cgroup2.devices.allow: c 236:* rwm
lxc.cgroup2.devices.allow: c 239:* rwm
lxc.mount.entry: /dev/nvidia0 dev/nvidia0 none bind,optional,create=file
lxc.mount.entry: /dev/nvidiactl dev/nvidiactl none bind,optional,create=file
lxc.mount.entry: /dev/nvidia-modeset dev/nvidia-modeset none bind,optional,create=file
lxc.mount.entry: /dev/nvidia-uvm dev/nvidia-uvm none bind,optional,create=file
lxc.mount.entry: /dev/nvidia-uvm-tools dev/nvidia-uvm-tools none bind,optional,create=file
bash
pct stop 100 && pct start 100

Step 3 — Install Driver Inside LXC

bash — inside lxc ct-100
wget https://us.download.nvidia.com/XFree86/Linux-x86_64/580.142/NVIDIA-Linux-x86_64-580.142.run
chmod +x NVIDIA-Linux-x86_64-580.142.run
./NVIDIA-Linux-x86_64-580.142.run --no-kernel-module  # host handles kernel module

Step 4 — NVIDIA Container Toolkit

bash — inside lxc ct-100
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg

curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

apt update && apt install -y nvidia-container-toolkit
nvidia-ctk runtime configure --runtime=docker --set-as-default
systemctl restart docker
⚠️
The driver version inside the LXC must match the host driver version exactly. Both must be 580.142.

GPU Object Detector (ONNX)

TensorRT is no longer supported on amd64. The ONNX detector with the stable-tensorrt Frigate image is used instead. NVIDIA GPUs are automatically detected when configured correctly.

Build YOLOv9 ONNX Model

bash
docker build . --build-arg MODEL_SIZE=t --build-arg IMG_SIZE=320 \
  --output /root/config/model_cache -f- <<'EOF'
FROM python:3.11 AS build
RUN apt-get update && apt-get install --no-install-recommends -y cmake libgl1 \
  && rm -rf /var/lib/apt/lists/*
COPY --from=ghcr.io/astral-sh/uv:0.10.4 /uv /bin/
WORKDIR /yolov9
RUN git clone https://github.com/WongKinYiu/yolov9.git .
RUN uv pip install --system -r requirements.txt
RUN uv pip install --system onnx==1.18.0 onnxruntime onnx-simplifier==0.4.* onnxscript
RUN uv pip install --system torch==2.5.1 torchvision \
  --index-url https://download.pytorch.org/whl/cpu
ARG MODEL_SIZE
ARG IMG_SIZE
ADD https://github.com/WongKinYiu/yolov9/releases/download/v0.1/yolov9-${MODEL_SIZE}-converted.pt \
  yolov9-${MODEL_SIZE}.pt
RUN sed -i "s/map_location='cpu'/map_location='cpu', weights_only=False/g" models/experimental.py
RUN python3 export.py --weights ./yolov9-${MODEL_SIZE}.pt --imgsz ${IMG_SIZE} --simplify --include onnx
FROM scratch
COPY --from=build /yolov9/yolov9-${MODEL_SIZE}.onnx /yolov9-${MODEL_SIZE}-${IMG_SIZE}.onnx
EOF
⚠️
torch==2.5.1 is pinned to avoid the weights_only=True breaking change in PyTorch 2.6. Do not upgrade without testing.

NVIDIA Driver Maintenance

Proxmox automatically updates its Linux kernel. When the kernel updates, the NVIDIA driver must be rebuilt — otherwise /dev/nvidia* devices disappear and Frigate stops working.

Symptoms of Kernel/Driver Mismatch

error
modprobe: FATAL: Module nvidia not found in directory /lib/modules/X.X.XX-pve
ls: cannot access '/dev/nvidia*': No such file or directory
Error response from daemon: not a device node

Fix After Kernel Update

bash — proxmox host
# Install headers for new kernel
apt install -y dkms pve-headers-$(uname -r)

# Rebuild driver for new kernel
dkms autoinstall -k $(uname -r)

# If status shows 'built' not 'installed', force it
dkms install nvidia/580.142 -k $(uname -r) --force

# Verify — must show 'installed' not 'built'
dkms status

# Load modules and verify devices
modprobe nvidia && modprobe nvidia-uvm && modprobe nvidia_modeset
ls -la /dev/nvidia*
nvidia-smi

DKMS Status Reference

StatusMeaningAction
installedReady to loadNone
builtCompiled but NOT installed in kernel treedkms install nvidia/580.142 -k $(uname -r) --force
not foundNot built for this kerneldkms autoinstall -k $(uname -r)

Auto-load Modules on Boot

bash
echo -e "nvidia\nnvidia-uvm\nnvidia_modeset" > /etc/modules-load.d/nvidia.conf
systemctl restart systemd-modules-load

# Auto-rebuild on kernel update via APT hook
echo 'DPkg::Post-Invoke {"dkms autoinstall 2>/dev/null || true";};' \
  > /etc/apt/apt.conf.d/99nvidia-dkms
⚠️
GTX 1050 Ti maximum driver is 580.142 — NVIDIA dropped Pascal (GTX 10 series) support in driver 590+.

GPU Detection Verification

A common issue is Frigate defaulting to CPU detection even when a GPU is present, causing high CPU usage and slow detection. Use these commands to confirm GPU detection is working.

Check ONNX Execution Providers

bash
docker exec -it frigate-cctv python3 -c "
import onnxruntime as ort
print('Providers:', ort.get_available_providers())
print('Device:', ort.get_device())
"
# Expected output:
Providers: ['TensorrtExecutionProvider', 'CUDAExecutionProvider', 'CPUExecutionProvider']
Device: GPU

Monitor GPU Usage

bash
watch -n 1 nvidia-smi   # GPU-Util should spike during detection events
⚠️
If GPU-Util shows 0% and no processes are listed, ONNX is using CPU. Add device: cuda to the detectors config and restart Frigate.

Shared Memory (SHM) Configuration

Shared memory is a fast memory space Frigate uses to pass video frames between processes. Think of it as a buffer — the more cameras you have, the bigger the buffer needs to be.

error
The current SHM size of 64.0MB is too small, recommend increasing it to at least 178MB.
yaml — docker-compose.yml
shm_size: "1024mb"     # 1GB for 2x 1080p cameras
bash — verify
docker compose down && docker compose up -d
docker logs frigate-cctv 2>&1 | grep -i "shm\|memory"

Adding Additional Cameras

Each camera needs two RTSP streams: a main high-quality stream for recording and a substream for detection. Both go through go2rtc before reaching Frigate.

Steps to Add a Camera

  • 1
    Check RTSP pathsLog into the camera's admin page and find the main and sub stream RTSP URLs.
  • 2
    Add to go2rtc streamsRegister both streams in the go2rtc section. Handle audio with #audio=aac or #audio=disable.
  • 3
    Add camera blockAdd the camera under cameras in config.yml. Start with fps: 5 for detect.
  • 4
    Increase resourcesIncrease shm_size and tmpfs size for each additional camera.
  • 5
    Restart and verifyRestart Frigate and check go2rtc streams: wget -q -O- http://localhost:1984/api/streams

Storage Estimation

Continuous recording storage depends on your camera's stream bitrate. With 2 cameras, multiply these figures by 2. Use this to decide how many days of footage your 1TB storage can hold.

BitratePer Day7 Days30 Days
2 Mbps~21 GB~150 GB~630 GB
4 Mbps~43 GB~300 GB~1.25 TB

💡
With 2 cameras on 1TB and 30-day retention, approximately 3 Mbps per camera is sustainable. Adjust continuous: days based on your actual stream bitrate.

Timezone Configuration

By default, Docker containers run in UTC. All recording timestamps, detection alerts, and review events will show the wrong time unless you configure the timezone. For the Philippines, set Asia/Manila (UTC+8).

Fix 1 — Environment Variable (Recommended)

yaml — docker-compose.yml
environment:
  - TZ=Asia/Manila

Fix 2 — Mount Host Timezone Files

yaml — docker-compose.yml
volumes:
  - /etc/localtime:/etc/localtime:ro
  - /etc/timezone:/etc/timezone:ro

Set Timezone on Proxmox Host

bash
timedatectl set-timezone Asia/Manila
docker compose down && docker compose up -d
docker exec -it frigate-cctv date   # should show UTC+8

Backup Procedures

Regular backups protect against accidental deletion, corruption, and hardware failure. The most critical files are config.yml and frigate.db.

What to Back Up

ComponentLocationPriorityFrequency
config.yml~/config/config.ymlCriticalEvery change
docker-compose.yml~/docker-compose.ymlCriticalEvery change
frigate.db~/frigate.dbHighDaily
Recordings/media/records/recordings/MediumContinuous
LXC Config/etc/pve/lxc/100.confCriticalEvery change
Zabbix Config/etc/zabbix/zabbix_agentd.confMediumEvery change

Automated Daily Backup Script

bash — ~/backup_frigate.sh
#!/bin/bash
BACKUP_DIR=~/backups/$(date +%Y%m%d)
mkdir -p $BACKUP_DIR
cp ~/config/config.yml $BACKUP_DIR/
cp ~/docker-compose.yml $BACKUP_DIR/
docker compose stop
cp ~/frigate.db $BACKUP_DIR/frigate_$(date +%Y%m%d).db
docker compose start
find ~/backups -type d -mtime +7 -exec rm -rf {} +
echo "Backup completed: $BACKUP_DIR"
bash — schedule with cron
chmod +x ~/backup_frigate.sh
crontab -e
# Add this line:
0 2 * * * /root/backup_frigate.sh >> /var/log/frigate_backup.log 2>&1

Restore from Backup

bash
cp ~/backups/20260501/config.yml ~/config/config.yml
docker compose stop
cp ~/backups/20260501/frigate_20260501.db ~/frigate.db
docker compose start
⚠️
Always use docker compose stop (not down) before restoring the database to avoid corruption.
